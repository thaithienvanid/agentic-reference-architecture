## Atomic transition commit

```json
{
  "contractVersion": "1.0.0",
  "runId": "run_01K2A",
  "expectedStateVersion": 27,
  "expectedRunSequence": 141,
  "workerLeaseId": "worker_lease_01K2L",
  "fencingToken": 42,
  "events": [
    {
      "eventType": "effect.reconciled",
      "schemaVersion": "1.0.0",
      "payloadDigest": "sha256:effect-reconciled"
    }
  ],
  "nextStateRef": "state://run_01K2A/version/28",
  "effectMutations": ["effect_01K2B:succeeded"],
  "invocationMutations": ["invocation_01K2I2:succeeded"],
  "approvalMutations": [],
  "budgetMutations": ["budget_reservation_01:reconciled"],
  "outboxEventIds": ["integration_evt_01"]
}
```

The implementation verifies expected state version, run sequence, and fencing token, then commits journal, projection, ledgers, and outbox atomically.

## Event categories

- Domain events carry business meaning.
- Execution events describe run, activity, effect, invocation, approval, child-link, and lease lifecycles.
- Integration events cross bounded contexts through the outbox.
- Audit events carry authoritative security and governance evidence.
- Telemetry signals support operations and may be sampled.

## Lifecycle authority

Status values and legal transitions are defined in [Lifecycle contracts](/reference/lifecycles). Historical events are immutable. Compatible evolution uses explicit contract/schema versions and readers or upcasters.
