# Implementation roadmap and variants

## Stage 0 — Architecture foundation

Build terminology, the run state machine, effect/error model, event envelope, threat model, one high-risk scenario, and port contract harness.

**Exit:** the team can identify the owner of every state transition and external effect.

## Stage 1 — One production agent

Build immutable versions, start/query/cancel APIs, model/tool gateways, SQL run journal, artifact store, basic policy, telemetry mapping, deterministic tests, and adapter contracts.

**Exit:** every model/tool call is attributable, budgeted, and auditable.

## Stage 2 — Durable execution

Add durable engine adapter, approvals, timers/events, effect ledger, reconciliation, checkpoints, resume, evidence replay, loop detection, and hard budgets.

**Exit:** a run survives worker loss and multi-day suspension without repeating completed effects.

## Stage 3 — Internal platform

Add catalogs, deployment compiler, promotion, shared policy/model gateways, evaluation service, workspace isolation, developer portal/SDK, and shared connectors.

**Exit:** several teams share the platform without sharing domain code or provider SDKs.

## Stage 4 — Enterprise SaaS

Add regional cells, isolation tiers, SSO/RBAC/ABAC, BYOM/BYOS, private networking, usage ledger/billing, tenant migration, SLOs, and disaster recovery.

**Exit:** a tenant can move from shared to dedicated execution without changing its agents.

## Stage 5 — Marketplace

Add publisher identity, package manifests, exact locks, signatures/provenance/SBOM, security/evaluation review, installation grants, upgrade/rollback/revocation, licensing, and revenue sharing.

**Exit:** third-party content can be installed and revoked without direct access to platform internals.

## Variant A — Agentic application

| Dimension | Recommendation |
|---|---|
| Scope | One product team and a small number of agents |
| Components | Modular app, execution kernel, worker, SQL journal, object store, model/tool gateways, basic policy/OTel |
| Omit | Public marketplace, global control plane, complex isolation and commerce |
| Deployment | One region and shared infrastructure |
| Main risks | Provider coupling, weak replay, unsafe tools |
| Team | Roughly 4–8 strong backend/AI engineers plus security support |
| Next step | Extract runtime and catalogs; add shared evaluation/policy |

## Variant B — Internal agent platform

| Dimension | Recommendation |
|---|---|
| Scope | Multiple internal product teams |
| Components | Shared control plane, regional cells, catalogs, model/tool/policy gateways, evaluation, developer portal |
| Omit initially | External package commerce and public publishers |
| Main risks | Central bottleneck, over-generic APIs, platform owning domain correctness |
| Team | Roughly 10–25 across platform, SRE, security, evaluation, DX |
| Next step | Add external tenancy, billing, publisher governance, dedicated cells |

## Variant C — Enterprise SaaS and marketplace

| Dimension | Recommendation |
|---|---|
| Scope | External tenants, regulated workloads, publishers, billing |
| Components | Regional/global control, shared/dedicated cells, BYOM/BYOS, private networking, marketplace, commerce |
| Main risks | Cross-tenant leakage, supply chain, billing disputes, regional consistency |
| Team | Multiple specialized teams; often 30+ across the complete capability set |
| Rule | Do not build this before runtime journal, policy boundary, and evaluation are reliable |

## Build versus buy

| Capability | Default posture |
|---|---|
| Domain/execution contracts | Build |
| Durable workflow engine | Adopt/managed |
| Policy evaluation engine | Adopt behind platform contract |
| Identity, secrets, KMS | Integrate mature services |
| Sandbox foundation | Adopt strong container/userspace-kernel/microVM technology |
| OpenTelemetry pipeline | Adopt |
| Evaluation frameworks | Integrate, but own datasets/gates/results |
| Marketplace domain and capability grants | Build |
| Billing attribution | Build core ledger; integrate invoicing |
| Vector/search infrastructure | Adopt as derived indexes |

## Main unresolved risks

- Provider model drift and imperfect reproducibility.
- Evaluation-judge calibration and correlated errors.
- Privacy deletion versus replay evidence.
- Policy behavior under control-plane outage.
- Workflow-engine migration and replay compatibility.
- Marketplace packages with context-triggered malicious behavior.
- Synthetic-persona validity and stereotype amplification.

Treat these as explicit research and operational workstreams, not hidden implementation details.
