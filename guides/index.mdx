|---|---|
| Product architect | Adopt ARA → domain/workflows → state/context/memory → worked example → checklist |
| Runtime engineer | Execution specification → durable runtime guide → interfaces → adapter guide → resilience testing |
| Provider integrator | Reference interfaces → adapter guide → conventions/events → contract tests |

## Documentation modes

| Need | Use |
|---|---|
| Follow a practical adoption sequence | [Adopt ARA](/guides/adoption) and worked examples |
| Complete a specific implementation or operations task | How-to guides and implementation pages |
| Understand alternatives and trade-offs | Explanation guides, ADRs, and research |
| Look up an exact term, interface, event, or evidence expectation | [Reference](/reference/index) |
| Determine mandatory behavior | [ARA specification](/specification/index) |
| Prepare a technical conformance claim | [Conformance checklist](/reference/conformance-checklist) |

## Guide contract

A how-to guide states an outcome, prerequisites, ordered actions or decisions, validation criteria, and failure or avoidance guidance. An explanation guide develops a mental model and trade-offs. Both link to the canonical specification or reference owner and avoid maintaining a second complete definition.
