# Runtime implementation guide

> **Status: Informative guide.** Normative ownership and semantics are in [RFC — Runtime and kernel](/rfc/runtime).

## Recommended composition

```text
Execution Kernel library
+ Runtime Service
+ Scheduler
+ Runtime Workers and WorkerLeases
+ DurableExecutionPort adapter
+ State Store and Platform Run Journal
+ Effect Ledger
+ Model/Tool/Policy/Secret/Sandbox gateways
```

## Kernel versus runtime

The `ExecutionKernel` is pure and deterministic. It evolves state and emits execution intents. The `RuntimeService` handles admission, persistence, scheduling, capacity, cancellation, streaming, and recovery.

A durable backend supplies timers, signals, suspension, and backend recovery. Its history is not the public ARA run model.

## Worker ownership

A worker acquires a `WorkerLease` with a fencing token. Worker loss followed by durable resume creates a new lease, not a new activity attempt.

A new `ActivityAttempt` is created only when the complete activity restarts from its declared boundary.

## Effects and invocations

```text
ActivityRun
└── Effect
    ├── Invocation 1
    └── Invocation 2
```

A transport retry, poll, or reconciliation query creates another `Invocation`. A changed prompt, context, arguments, tool, or objective creates another `Effect`.

## Retry table

| Situation | Response |
|---|---|
| Pure task fails before durable output | Retry through `ActivityAttempt` when safe |
| Provider returns 429 | New `Invocation` after durable backoff |
| Mutation response is lost | Reconcile the same `Effect` |
| Worker dies after checkpoint | New `WorkerLease`, resume same `ActivityRun` |
| Deliberation continues with new evidence | New `Iteration` |
| Experiment repeats from frozen baseline | New `ExperimentTrial` referencing a `WorkflowRun` |

## Runtime invariants

- One authoritative state-transition writer per run.
- Effect intent is durable before the first invocation.
- Idempotency keys remain stable across invocations.
- Stale leases cannot commit.
- Waiting releases active workers.
- Completed effects are reused during resume and evidence replay.
- Cancellation does not erase completed mutations.
- Fallback routes are declared in the deployment snapshot.
