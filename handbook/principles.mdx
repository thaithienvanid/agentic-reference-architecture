# Principles and invariants

> **Status: Normative.** This page is the canonical list of architecture invariants. Summaries elsewhere must link here rather than define competing rule sets.

## Deterministic shell, nondeterministic leaves

Model invocations, retrieval, and external tools are effects. They may produce uncertain results. The lifecycle surrounding them must be explicit and deterministic.

```typescript
interface RunMachine {
  evolve(state: RunState, event: RunEvent): RunState;
  decide(state: RunState): readonly EffectIntent[];
}
```

For a fixed state and event sequence, `evolve` must return the same state.

## Core invariants

1. Published definitions are immutable.
2. Every run pins exact versions and content digests in a deployment snapshot.
3. A model result cannot directly mutate authoritative domain state.
4. Every effect has a stable identity, declared capability, and side-effect classification.
5. Every concrete provider/adapter call is an invocation attempt associated with exactly one effect.
6. Irreversible effects require idempotency or reconciliation.
7. Human approval binds to the exact action, arguments digest, tool version, policy version, approver, and expiry.
8. Tenant identity is present at every state, cache, event, artifact, trace, credential, and usage boundary.
9. Child agents receive an explicit capability subset; authority does not expand implicitly.
10. Workflow state, conversation, working context, memory, checkpoints, traces, and artifacts remain separate.
11. Activity outputs are validated and committed before they become downstream inputs.
12. Trials, iterations, branches, activity retries, and invocation retries have distinct semantics.
13. Cross-run chaining uses immutable artifacts and explicit decisions, never hidden shared memory.
14. Framework and provider types stop at adapters.
15. Evaluation contracts are part of deployable behavior.
16. Audit and billing do not depend solely on sampled telemetry.
17. Deterministic software is preferred whenever it is sufficient.

## Side-effect discipline

```text
propose -> validate -> authorize -> reserve budget -> approve if required
-> persist effect intent -> execute invocation attempt(s) -> record effect result
-> reconcile usage -> advance state
```

A timeout is not proof that an external mutation failed. The runtime must distinguish definite success, definite failure, and an ambiguous outcome.

## Explicit contracts over agent magic

Every activity should declare input/output schemas, required capabilities, policies, budgets, timeout/retry behavior, side-effect constraints, evaluation expectations, and completion/failure semantics.

## Small core, strong guidance

The platform core defines reusable execution concepts. Domain-specific concepts such as `Study`, `Participant`, `Claim`, or `ProcurementCase` belong in bounded contexts that map onto core primitives.
