# Documentation architecture

> **Status: Project governance.** This page governs the documentation set; it does not add runtime requirements.

## Goal

ARA documentation must let a reader answer six questions without searching through competing complete explanations:

| Question | Primary destination |
|---|---|
| What does ARA require? | **Specification** |
| How do I adopt, implement, integrate, or operate it? | **Guides** |
| What is the exact term, field, interface, schema, profile, or evidence expectation? | **Reference** |
| Why is the architecture shaped this way? | Explanation guides, ADRs, and research |
| Which reusable composition should I apply? | **Patterns** |
| What does a complete system look like? | **Worked examples** |

## Research basis

The documentation model combines:

- Diátaxis separation of learning, task-oriented how-to, exact reference, and explanation.
- RFC 7322 and BCP 14 discipline for clear technical publication and normative language.
- Google developer-documentation guidance for direct, consistent, project-specific prose.
- Mintlify tabs, groups, roots, landing pages, and redirects for reader navigation.

ARA adds a separate normative specification layer because conformance requirements do not belong inside an ordinary tutorial, guide, or reference profile.

## Authority order

```text
Normative specification
  -> normative reference contracts
    -> accepted ADRs
      -> informative guides
        -> patterns and worked examples
          -> research and project review history
```

A lower-authority page may explain, operationalize, or illustrate a higher-authority rule. It must not redefine it.

| Content class | Authority | Primary job |
|---|---|---|
| Specification | Normative | Semantics, invariants, ownership, profiles, conformance |
| Normative reference | Normative when marked | Exact required names, interfaces, schemas, and representations |
| Reference profile | Optional until declared | Concrete REST, manifest, or interoperability profile |
| Reference checklist | Assessment aid | Map requirements to evidence and review status |
| ADR | Decision record | Rationale, alternatives, consequences, revisit conditions |
| How-to guide | Informative | Complete and verify one task |
| Explanation guide | Informative | Explain mental models, alternatives, and trade-offs |
| Pattern | Informative | Reusable composition with use/avoid guidance |
| Worked example | Informative | End-to-end illustration; no new requirements |
| Research | Evidence | Facts, inferences, recommendations, and sources |
| Review history | Historical | Consolidated decisions and remediation; not current authority |

## Reader-facing architecture

The site has five top-level journeys:

1. **Specification** — normative semantics and cumulative profiles.
2. **Guides** — adoption, application design, runtime integration, security, and operations.
3. **Reference** — exact contracts, checklist, optional profiles, and diagrams.
4. **Patterns & examples** — reusable compositions and complete scenarios.
5. **Project** — decisions, releases, roadmap, research, review history, and maintenance.

A new top-level tab requires evidence that an important reader job cannot be served by a landing page or navigation group.

## Repository map

```text
specification/  normative semantic specification and conformance
reference/      exact contracts, glossary, checklist, profiles, and diagrams
guides/         adoption and reader pathways
handbook/       application, runtime, integration, security, and operations guides
evaluation/     evaluation-driven development guides and adapters
implementation/ module layout, rollout, testing, and resilience
patterns/       reusable composition patterns
examples/       end-to-end worked systems
decisions/      ADRs and decision guidance
research/       primary evidence and comparisons
project/        documentation, release, and review governance
```

`specification/` is the single semantic authority. `reference/` defines exact machinery and representations. `handbook/`, `guides/`, `evaluation/`, and `implementation/` explain how and why to apply them. The repository does not maintain an RFC-named, standards-claiming, or second specification tree.

Physical folder names are implementation details below the authority model. Rename them only when reader or maintainer benefit exceeds redirect, compatibility, history, and review cost.

## Documentation-mode mapping

| Reader need | Location |
|---|---|
| Follow a practical adoption path | `guides/adoption.mdx` and worked examples |
| Complete a design or operations task | How-to and implementation guides |
| Look up exact machinery | Reference pages and checklist |
| Understand alternatives and trade-offs | Explanation guides, ADRs, and research |
| Determine mandatory conformance | Specification and normative reference contracts |

A page normally serves one primary mode. Cross-link rather than combining tutorial, procedure, lookup, rationale, and end-to-end example into one book-sized page.

## Reader journeys

| Reader | Entry point | Expected result |
|---|---|---|
| Product architect | Adoption guide | Minimum justified profile and architecture-review package |
| Runtime engineer | Execution and runtime specification | Recoverable runtime design |
| Provider integrator | Reference interfaces and adapter guide | Qualified provider adapter |

## Page contracts

Every public MDX page declares `title` and `description` frontmatter and states authority when it could be ambiguous.

### Specification

```text
status and scope
terms or model
normative requirements and ownership
failure and security implications
conformance consequences
links to exact reference contracts
```

Use uppercase BCP 14 terms only for normative meaning. Keep tutorials and vendor integration procedures elsewhere.

### Normative reference

```text
status and scope
canonical fields, interfaces, schemas, or names
constraints and compatibility
compact mechanically checkable examples
```

Optimize for lookup. Move rationale to guides or ADRs and version representation changes explicitly.

### Reference profile or checklist

Profiles map concrete representations to normative semantics and become binding only when declared. Checklists map requirements to evidence and closed statuses; they cannot weaken or strengthen the canonical owner.

### How-to guide

```text
outcome
prerequisites
ordered decisions and actions
failure and security handling
validation
avoidance or next steps
```

A component inventory without an executable procedure is explanation, not a how-to.

### Explanation guide

```text
problem or question
mental model
alternatives and trade-offs
failure, security, and operational implications
decision guidance
canonical links
```

### Pattern

```text
intent
use when / avoid when
structure and participants
state, failure, security, and cost behavior
evaluation and observability
concrete example
```

A pattern composes existing semantics; it does not add a universal core type for every technique.

### Worked example

```text
scenario and requirements
exact versions and assumptions
execution trace
state, effect, invocation, approval, and evidence mapping
failure and recovery
evaluation and limitations
```

### Research and review history

Research distinguishes fact, inference, and ARA recommendation; cites primary sources; and records retrieval dates. Review history consolidates scope, decisions, remediation, validation status, and the current canonical owner.

## Duplication review

Before retaining a page, ask:

1. Does it answer a different reader question from the canonical owner?
2. Does it add procedure, rationale, lookup value, evidence mapping, pattern, or a complete example?
3. Can copied definitions be replaced with links without reducing usefulness?
4. Can the canonical owner change without requiring reviewers to remember another complete version?
5. Is authority clear from the title and first screen?
6. Does navigation match the page’s primary job?

If not, merge useful material into the owner or a purposeful guide, redirect the old URL, and remove the duplicate.

## Content budgets

| Page type | Default unit |
|---|---|
| Specification | One semantic boundary or profile concern |
| Normative reference | One coherent contract family |
| Checklist | One cumulative assessment path |
| How-to | One verifiable reader outcome |
| Explanation | One mental model or decision cluster |
| Pattern | One reusable composition |
| Worked example | One complete scenario |
| Landing page | Entry choices and minimal orientation |

A page is long enough to complete its job and no longer. Do not split merely to increase page count.

## Change propagation

A semantic change is complete only when affected artifacts agree:

```text
specification rule
-> normative reference or schema
-> conformance checklist and tests
-> ADR when consequential
-> guides and decision tables
-> diagrams
-> examples and manifests
-> evaluation gates and datasets
-> compatibility notes, release record, and redirects
-> deterministic documentation checks
```

Classify each change as semantic, representation/profile, procedural, explanation/editorial, navigation/redirect, research/evidence, or project history.

## Navigation rules

- Keep the five top-level tabs stable without reader evidence for change.
- Give each major section a landing page.
- Use task-oriented group labels and short page lists.
- Put a page where readers seek its primary job, not where a team owns code.
- Do not list a page in multiple navigation locations; cross-link instead.
- Navigate every public page or explicitly mark it hidden/draft.
- A redirect source must not shadow a live page.
- Preserve moved URLs through compatibility review.

## Quality model

| Dimension | Review question |
|---|---|
| Correctness | Does content match canonical semantics and current primary sources? |
| Findability | Can the intended reader reach it and the next task quickly? |
| Authority clarity | Is normative, reference, procedural, explanatory, or historical status obvious? |
| Completeness | Can the page finish its stated job, including failure and validation? |
| Economy | Is it free of competing definitions and unnecessary repetition? |
| Reproducibility | Are versions, commits, digests, evidence, dates, and assumptions explicit? |
| Maintainability | Can semantic changes propagate predictably and mechanically? |

## Content lifecycle and review triggers

```text
proposal -> draft -> reviewed -> candidate -> published
-> corrected or versioned -> deprecated -> redirected or archived
```

Review is triggered by semantic or terminology changes; interface, event, schema, or profile changes; stale provider or legal claims; failed page contracts; navigation moves; repeated reader confusion; new conformance profiles; and release baselines.

## Reader evidence and continuous improvement

Review search failures, repeated backtracking, broken inbound links, time to task, reader completion, support questions, weak onward navigation, accessibility findings, and conformance gaps. Analytics guide review but do not replace representative reader testing.

Before a stable major release, verify that representative readers can select a profile, locate effect-versus-invocation semantics, implement adapter failure handling, trace reconciliation, identify conformance evidence, distinguish optional profiles, and find publication status and errata.

## Publication and errata

Stable publications identify both semantic version and exact commit or digest. Editorial corrections, ambiguities, security defects, and semantic changes follow [Releases, versions, and errata](/project/releases-and-errata). Published history is not silently rewritten.

## Quality gates

Documentation CI verifies:

- Valid `docs.json` with the intended five tabs.
- Existing, unique navigation targets and group roots.
- No unintended orphan or duplicate public page.
- Required specification, reference, guide, checklist, and governance pages.
- Frontmatter and authority boundaries.
- Critical how-to and checklist contracts.
- No retired live RFC-named, audit, or competing specification tree.
- Unique redirects whose sources do not shadow live pages.
- Canonical terminology, hard gates, state transitions, money, and publication identity.
- Parseable JSON/YAML examples where mechanically checkable.
- Mintlify validation and internal links.

## Maintenance principle

> Prefer one authoritative definition plus several purposeful views over several nearly identical complete documents.
